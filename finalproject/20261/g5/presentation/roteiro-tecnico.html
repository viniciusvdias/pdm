Bloco 1 · Contexto

Por que recomendação é um problema de Big Data?

O problema não é difícil por causa do volume isolado — é difícil porque os dados são relacionais e interconectados: um usuário avaliou dezenas de filmes, cada filme pertence a múltiplos gêneros, e cada gênero conecta milhares de filmes.

"Imaginem tentar fazer uma query SQL para 'filmes que usuários similares a você gostaram, que pertencem a gêneros que você curte, e que possuem tags semanticamente relevantes'. Seriam 4 ou 5 JOINs em tabelas de milhões de linhas — inviável em tempo real."

"A solução que escolhemos foi o Neo4j — um banco de dados orientado a grafos que resolve exatamente esse problema com o conceito de index-free adjacency: para percorrer do usuário ao filme ao gênero a outro filme, o custo é O(1) por aresta, independente do tamanho total do banco."
Bloco 2 · Dataset

genome_2021 — O dataset

Dataset fornecido pelo professor via Google Drive. Nome técnico: MovieLens Tag Genome 2021, produzido pelo GroupLens Research da Universidade de Minnesota.

ArquivoFormatoCamposTamanho
metadata.jsonJSON arraymovieId, title, genres, year, imdbId, tmdbId17 MB
ratings.jsonNDJSON (1/linha)userId, movieId, rating, timestamp1,38 GB
tags.jsonJSON arraytagId, tag37 KB
tag_count.jsonJSON arraytagId, movieId, count9,3 MB
tagdl.csvCSVmovieId, tagId, score305 MB
ratings.json usa NDJSON (uma linha por registro JSON) porque 1,38 GB não pode ser carregado como um array inteiro na memória. O injector lê linha a linha com for line in f: json.loads(line).

Schema do grafo no Neo4j

(:User {userId}) └──[:RATED {rating, timestamp}]──>(:Movie {movieId, title, year}) ├──[:BELONGS_TO]──>(:Genre {name}) ├──[:HAS_TAG {score}]──>(:Tag {tagId, name}) └──[:TAG_COUNT {count}]──>(:Tag)

Por que grafo? A aresta [:HAS_TAG {score}] carrega o Tag Genome score — um valor entre 0 e 1 que representa a relevância semântica de uma tag para um filme, calculado por machine learning. Isso é o diferencial do dataset genome_2021 em relação ao MovieLens padrão.

Detalhe: sequência de boot

  1. db-neo4j sobe e passa no healthcheck (wget http://localhost:7474)
  2. injector-job aguarda Neo4j saudável → carrega todos os arquivos → encerra com exit 0
  3. api-gateway aguarda o injector completar (service_completed_successfully) → sobe e fica disponível
"Essa ordem é garantida pelo Docker Compose via depends_on com conditions. Sem isso, a API poderia subir antes dos dados estarem no banco."

Bloco 4 · Ingestão

injector-job — Como os dados entram no Neo4j

O injector é um script Python que roda uma única vez. Ele usa o driver oficial neo4j==5.19.0 e insere dados em lotes via UNWIND no Cypher — muito mais eficiente que inserções linha a linha.

Sequência de ingestão (6 etapas)

#EtapaFonteO que cria no grafo
1Constraints—UNIQUE em userId, movieId, genre.name, tagId
2Tagstags.json37 nós (:Tag)
3Filmes + Gênerosmetadata.json100 (:Movie) + 18 (:Genre) + [:BELONGS_TO]
4Avaliaçõesratings.json (NDJSON)150 (:User) + 1.500 [:RATED]
5Tag countstag_count.json300 [:TAG_COUNT {count}]
6Tag genome scorestagdl.csv500 [:HAS_TAG {score}]

Por que UNWIND é eficiente?

UNWIND $rows AS row
MERGE (m:Movie {movieId: toInteger(row.movieId)})
  SET m.title = row.title
WITH m, row
UNWIND split(row.genres, '|') AS genreName
MERGE (g:Genre {name: genreName})
MERGE (m)-[:BELONGS_TO]->(g)
"Em vez de fazer uma transação por registro, enviamos 500 registros de uma vez com UNWIND. O Neo4j processa tudo em uma única transação, reduzindo o overhead de rede e de commit."

Bloco 5 · Queries Cypher

Os 3 Workloads — Queries no grafo

WORKLOAD-1 — Recomendação por Gênero Compartilhado

Endpoint: GET /recommend?userId=1

Encontra filmes não vistos que pertencem aos mesmos gêneros dos filmes que o usuário avaliou com nota ≥ 4.0.

MATCH (u:User {userId: $uid})-[r:RATED]->(m1:Movie)-[:BELONGS_TO]->(g:Genre)
WHERE r.rating >= 4.0
MATCH (m2:Movie)-[:BELONGS_TO]->(g)
WHERE NOT (u)-[:RATED]->(m2)
WITH m2, collect(DISTINCT g.name) AS sharedGenres, count(*) AS score
RETURN m2.title, sharedGenres, score
ORDER BY score DESC LIMIT 10

WORKLOAD-2 — Filtragem Colaborativa

Endpoint: GET /recommend/collaborative?userId=1

Encontra usuários que avaliaram os mesmos filmes com notas altas e recomenda filmes que eles curtiram mas o alvo não viu.

MATCH (u:User {userId: $uid})-[r1:RATED]->(m:Movie)<-[r2:RATED]-(similar:User)
WHERE r1.rating >= 4.0 AND r2.rating >= 4.0 AND u <> similar
MATCH (similar)-[r3:RATED]->(rec:Movie)
WHERE NOT (u)-[:RATED]->(rec) AND r3.rating >= 4.0
WITH rec, count(DISTINCT similar) AS similarUsers, avg(r3.rating) AS avgRating
RETURN rec.title, similarUsers, round(avgRating, 2) AS avgRating
ORDER BY similarUsers DESC, avgRating DESC LIMIT 10

WORKLOAD-3 — Tag Genome (diferencial do dataset)

Endpoint: GET /recommend/tag-genome?userId=1

Usa os scores de relevância semântica do tagdl.csv: identifica as tags mais relevantes dos filmes favoritos do usuário e encontra outros filmes com alta relevância nessas mesmas tags.

MATCH (u:User {userId: $uid})-[r:RATED]->(watched:Movie)
WITH u, collect(watched.movieId) AS watchedIds
MATCH (u)-[r2:RATED]->(m:Movie)-[ht:HAS_TAG]->(t:Tag)
WHERE r2.rating >= 4.0 AND ht.score >= 0.5
WITH u, watchedIds, t, avg(ht.score) AS tagRelevance
ORDER BY tagRelevance DESC LIMIT 20
MATCH (rec:Movie)-[ht2:HAS_TAG]->(t)
WHERE NOT rec.movieId IN watchedIds AND ht2.score >= 0.5
WITH rec, count(DISTINCT t) AS matchedTags, avg(ht2.score) AS avgTagScore
RETURN rec.title, matchedTags, round(avgTagScore, 4)
ORDER BY matchedTags DESC, avgTagScore DESC LIMIT 10
"O WORKLOAD-3 é o que distingue este projeto de uma recomendação comum. Em vez de apenas contar avaliações, usamos scores computados por machine learning que medem o quanto cada tag é semanticamente relevante para cada filme — por exemplo, 'psychological thriller' pode ter score 0.95 para Memento e 0.12 para Toy Story."

Limitações

Conclusões

  • Neo4j com index-free adjacency resolve o problema de recomendação em grafos com latência submilisegundo por aresta
  • A arquitetura de microsserviços com Docker Compose garante reprodutibilidade total em qualquer máquina com Docker
  • O Tag Genome agrega dimensão semântica à recomendação — algo inviável em abordagens relacionais tradicionais
  • A ingestão em lotes via UNWIND + constraints UNIQUE garante idempotência: rodar o injector múltiplas vezes não duplica dados
  • Taxa de erro 0% em ~108k requisições demonstra robustez da solução mesmo com hardware modesto

Apêndice · Perguntas Frequentes

Possíveis perguntas da banca

❓ Por que Neo4j e não MongoDB ou PostgreSQL?

"MongoDB não tem travessia de grafos nativa. PostgreSQL com JOINs recursivos funciona, mas o custo é O(n·m) onde n=usuários e m=filmes. O Neo4j com index-free adjacency resolve cada aresta em O(1) — para 25 milhões de avaliações, a diferença é de segundos vs milissegundos."

❓ O que é NDJSON e por que ratings.json usa isso?

"NDJSON (Newline Delimited JSON) é um formato onde cada linha é um objeto JSON independente. O arquivo ratings.json tem 1,38 GB — se fosse um array JSON, teríamos que carregar o arquivo inteiro na memória antes de processar o primeiro registro. Com NDJSON, lemos e processamos linha a linha, com uso de memória constante."

❓ O que é o Tag Genome e por que ele é melhor que tags normais?

"Tags normais (como em tags.json) são subjetivas e esparsas — só aparecem se um usuário explicitamente aplicou a tag. O Tag Genome calcula, via machine learning, um score entre 0 e 1 para TODOS os pares (filme, tag), mesmo que nenhum usuário tenha aplicado a tag. Isso significa que podemos comparar filmes por similaridade semântica mesmo sem avaliações suficientes — resolve o cold-start problem parcialmente."

❓ Por que a latência máxima foi 951 ms no Run 1?

"O Neo4j roda sobre a JVM (Java Virtual Machine), que precisa de um período de aquecimento para compilar o bytecode em código nativo via JIT (Just-In-Time). O pico de 951 ms ocorreu nos primeiros segundos do Run 1. Nos Runs 2 e 3, com a JVM aquecida, o máximo foi abaixo de 210 ms. Em produção, containers Neo4j são mantidos rodando continuamente, então esse efeito não acontece."

❓ A taxa de erro foi 4% no Run 1 mas 0% nos outros. Por quê?

"Os 4,01% de erro no Run 1 correspondem às requisições de usuários para os quais o grafo não tinha recomendações (retornavam HTTP 404, não 500). O k6 contava 404 como falha no check, mas era comportamento esperado. Nos Runs 2 e 3, ajustamos o script para aceitar 404 como sucesso — por isso os Runs 2 e 3 mostram 0% de erro."

❓ O projeto escala para o dataset completo (1,38 GB)?

"O código está preparado: o injector lê ratings.json em NDJSON linha a linha (memória constante) e insere em lotes de 500. A única limitação prática é o tempo de ingestão e a RAM do Neo4j. Com 16 GB de RAM e heap de 8G para o Neo4j, estimamos ingestão de 25M avaliações em 30–60 minutos. As queries Cypher escalarão, mas latências serão maiores — o p90 pode passar de 100 ms com o dataset completo."
