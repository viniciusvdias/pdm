CONTEXTO
--------
O projeto foi desenvolvido como um conjunto de notebooks Jupyter (rodando dentro de
um contêiner Docker com GPU). Para a entrega, ele foi reorganizado na estrutura de
pastas exigida pelo template (gX/: bin, src, misc, datasample, presentation, README)
e adaptado para rodar "out of the box" apenas com Docker, sem intervencao manual.

Tema do projeto: provar, com evidencia estatistica, que o "Small File Problem"
(ler milhares de .jpg isolados) causa GPU Starvation num pipeline de ML, e que
converter as imagens para blocos .parquet via Spark (ETL) resolve o gargalo,
aumentando o throughput de ingestao (imagens/segundo).

================================================================================
 AJUSTES E MELHORIAS APLICADOS
================================================================================

1. NOTEBOOKS -> SCRIPTS PYTHON HEADLESS (src/)
   - Os 4 notebooks foram convertidos em .py que rodam sem Jupyter/intervencao.
   - Removidos os elementos que so funcionam no notebook: comandos de shell (!),
     "magics" (%), e auto-display de DataFrames na ultima linha da celula.
   - Graficos: trocado plt.show() por plt.savefig() + backend "Agg" (headless),
     salvando cada figura em arquivo .png separado.

2. CODIGO COMPARTILHADO EXTRAIDO (src/bench_common.py)
   - Os notebooks 02 e 04 duplicavam ~200 linhas identicas (monitor de recursos,
     leitura de limites via cgroup, construcao do modelo). Isso foi centralizado
     em bench_common.py, deixando cada benchmark so com o que e proprio dele.

3. PARAMETRIZACAO POR VARIAVEIS DE AMBIENTE
   - Caminhos e parametros do experimento (RAW_DIR, PARQUET_DIR, RESULTS_DIR,
     N_ROUNDS, BATCH_SIZE, N_PASSES, TARGET_GB...) agora vem de env vars, com
     defaults sensatos. Assim o MESMO codigo serve para o smoke test (datasample)
     e para o dataset completo, sem editar nada - so mudam as variaveis nos scripts bin/.

4. ORQUESTRACAO EM PROCESSOS SEPARADOS (src/run_all.py)
   - Roda 02 -> 03 -> 04 na ordem correta, cada etapa num processo novo. Isso
     evita que o cache de disco / estado do TensorFlow de uma etapa contamine a
     medicao da outra (benchmark mais justo).

5. EXECUCAO "OUT OF THE BOX" COM DOCKER (bin/)
   - bin/run.sh: build + pipeline completo sobre o datasample, com UM comando.
   - bin/run_full.sh: mesmo fluxo com o dataset completo (inclui a duplicacao).
   - bin/download_data.sh: obtencao/descompactacao do dataset.
   - Todos usam "docker compose -f misc/docker-compose.yml ...".

6. DOCKER ADAPTADO (misc/)
   - Volumes e contexto de build reorganizados para a nova estrutura de pastas.
   - Saidas geradas vao para ./output/ (montado no host, visivel; ignorado no git).
   - Mantido o limite de 2 vCPU (que e justamente o que forca o gargalo estudado)
     e o acesso a GPU.

7. AMOSTRA DE DADOS (datasample/)
   - 40 imagens (~862 KB), abaixo do limite de 1 MB do template.
   - Serve para provar que o projeto RODA. Aviso importante: com poucas imagens o
     dataset cabe na memoria/cache e o Small File Problem NAO aparece - por isso os
     numeros do smoke test sao inconclusivos de proposito. Os resultados validos do
     relatorio vem do dataset completo (bin/run_full.sh).

8. HIGIENE DE REPOSITORIO (.gitignore)
   - Ignora output/ (resultados gerados em runtime) e os arquivos de contexto
     pessoal do desenvolvimento (CLAUDE.md, .claude/, agent_docs/).


================================================================================
 COMO RODAR
================================================================================

Pre-requisito: Docker + Docker Compose, com suporte a GPU NVIDIA (nvidia-container-toolkit).

A) SMOKE TEST (rapido, prova que roda - numeros inconclusivos):
     ./bin/run.sh
   Resultados em: ./output/results/

B) DATASET COMPLETO (reproduz os resultados do relatorio):
     1. Baixe o "train.zip" de https://www.kaggle.com/c/dogs-vs-cats/data
        e coloque em ./output/data/raw_jpg/train.zip
     2. ./bin/download_data.sh    (descompacta)
     3. ./bin/run_full.sh         (preparo + duplicacao + benchmarks)
   Resultados em: ./output/results/

C) MODO INTERATIVO (inspecionar os notebooks):
     docker compose -f misc/docker-compose.yml up
   Depois abrir o JupyterLab em http://localhost:8888
